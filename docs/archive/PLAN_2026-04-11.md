## Part 1 — Walkthrough: what happens when you "upload a video"

### Agents, in order

1. **`ingest`** — Copies source files from SD card / given path into `episodes/<ep_id>/source/`. ffprobes the first video file and stores `source_properties` (width, height, codec, pix_fmt, fps, color_space) into `episode.json`. If `--audio-path` was provided (H6E), also copies those .wav tracks into `audio_tracks`.
2. **`stitch`** — Runs ffmpeg concat-demuxer to merge the source .MP4 files into `source_merged.mp4`. Deletes the per-file copies from `source/`.
3. **`audio_analysis`** — Extracts L and R channels from `source_merged.mp4` into `work/left.wav`, `work/right.wav`, `work/left_channel.npy`, `work/right_channel.npy`. Computes channel correlation + RMS delta. Classifies as `true_stereo` (two distinct mics per channel) or `audio_channels_identical` (mono duplicated).
4. **PAUSE 1 — `awaiting_crop_setup`**: Pipeline blocks. The user must go to the web UI → Edit Crop → define speaker count, per-speaker framing (center point + zoom), and (if H6E) per-speaker audio track assignment. UI POSTs to `/api/episodes/{id}/crop-config`. The endpoint also deletes cached artifacts for any crop-dependent agent, transitions status back to `processing`, and auto-generates `work/audio_mix.wav` (via `lib/audio_mix.py`, with config so enhancement runs).
5. **`speaker_cut`** — Reads `left_channel.npy`/`right_channel.npy` (for L/R mode) OR per-mic H6E files (for N-speaker mode). Detects which speaker is talking at each frame, produces `segments.json` with per-segment speaker labels (`speaker_0`, `speaker_1`, ..., or `BOTH` for overlap) and per-segment duration. Also writes `speaker_N_rms_db.npy` for each speaker.
6. **`transcribe`** — Deepgram REST API (we bypass their SDK v5). Produces `transcript.json` (raw) and `diarized_transcript.json` (utterances with word-level timestamps + speaker labels).
7. **`clip_miner`** — Claude Opus picks 10 virality-ranked clips from the diarized transcript → `clips.json`. Episode directory is **renamed** here to include the guest name slug.
8. **`longform_render`** (depends on `speaker_cut` + `transcribe`):
   - Re-generates `work/audio_mix.wav` via `lib/audio_mix.py:generate_audio_mix()`.
     - H6E path: mixes multi-track .wav files per the user's crop config, applies sync offset, outputs dual-mono stereo.
     - **Camera-audio path** (`_generate_camera_audio_mix`): extracts audio from `source_merged.mp4`, **sums L+R to mono** (just fixed — was bug), outputs dual-mono stereo WAV.
     - Either path then runs through `lib/audio_enhance.py:enhance_audio()` → DFN3 (wet/dry mix 0.5) → afftdn → adeclick → highpass → lowpass → acompressor → deesser → two-pass loudnorm -14 LUFS.
   - For each segment from `speaker_cut`: ffmpeg seeks `source_merged.mp4`, applies `lut3d → crop → scale+lanczos → format=yuv420p → hqdn3d → cas → eq → subtitles`, encodes video-only via `h264_videotoolbox` (q:v 65, profile high) → `longform_seg_XXXX.mp4`.
   - Concats all video segments with stream-copy → `longform_raw.mp4`.
   - Muxes `longform_raw.mp4` + `audio_mix.wav` → `longform.mp4` with `-c:v copy -c:a aac -shortest -use_editlist 0 -movflags +faststart`. Audio is encoded **once**, not per-segment, to avoid AAC frame-padding drift.
9. **PAUSE 2 — `awaiting_longform_approval`**: User reviews longform.mp4, then POSTs `/api/episodes/{id}/approve-longform`. Approval triggers the rest of the pipeline.
10. **`shorts_render`** (depends on `clip_miner` + `speaker_cut`): For each clip, two-step render: (1) video-only with precise `trim=/setpts` filter, (2) mux audio slice from `audio_mix.wav`. Same color pipeline as longform. Outputs to `shorts/clip_NN.mp4`.
11. **`metadata_gen`** — Claude Sonnet generates per-platform metadata (YouTube/TikTok/Instagram/X/LinkedIn/Threads/Pinterest/Bluesky/Facebook). Stored in `metadata/metadata.json`.
12. **`thumbnail_gen`** — OpenAI image model generates a caricature thumbnail from the guest's face frame.
13. **`qa`** — Validates all outputs (durations, A/V sync, file sizes) → `qa/qa.json`.
14. **`podcast_feed`** — Generates podcast RSS feed with MP3 audio (from audio_mix.wav → lame encode).
15. **PAUSE 3 — `awaiting_publish_approval`** (added today): User must POST `/api/episodes/{id}/approve-publish`. Publish agent has a safety check that refuses to run without `publish_approved = True` in episode.json.
16. **`publish`** — Submits videos to Upload-Post API (curl). Each clip can be immediate or scheduled. Honors per-platform `enabled` flags in config.
17. **PAUSE 4 — `awaiting_backup_approval`**: User POSTs `/api/episodes/{id}/approve-backup` to trigger final step.
18. **`backup`** — rsyncs the episode dir to the Seagate drive, then optionally clears the SD card.

### Upload permutations

| Case | `source_path` | `audio_path` | Audio source | speaker_cut mode |
|---|---|---|---|---|
| **A) DJI only, stereo split mics** | camera SD or folder | *empty* | L/R channels from camera .MP4 | L/R (true_stereo) |

## Part 2 — Known bugs (ranked by severity)

### 🔴 BUG-1: Camera audio L/R not merged (JUST FIXED)

**Symptom**: "It's super obvious you didn't parse the two separate mics separately — it's playing left and right."
**Cause**: `lib/audio_mix.py:_generate_camera_audio_mix()` was extracting raw stereo from `source_merged.mp4` with no channel merge. Each DJI Mic Mini wrote to a separate channel of the camera's stereo audio, so the resulting `audio_mix.wav` had speaker 0 only on L and speaker 1 only on R.
**Evidence**: L/R correlation on EP2 audio_mix.wav = **-0.016** (should be +1.0 for dual-mono). L RMS 1417, R RMS 6837 (5x ratio).
**Fix applied**: Added `-af "pan=stereo|c0=0.5*c0+0.5*c1|c1=0.5*c0+0.5*c1"` to the extraction ffmpeg command. Also updated docstring — the incorrect comment claimed speaker_cut needed the raw split, but speaker_cut actually reads `left_channel.npy`/`right_channel.npy` which are produced separately by `audio_analysis`.
**Status**: Code fixed. Both EP1 and EP2 still need to be re-rendered to pick up the fix. The stale `audio_mix.wav` has been deleted from EP2; EP1 was already deleted.

### 🔴 BUG-2: Shorts audio/video desync on EP1 (NOT YET FIXED)

**Symptom**: User reported EP1 shorts came out with audio and video completely unsynced.
**Likely cause**: `agents/shorts_render.py:_render_short()` uses `-use_editlist 0` on the final mux step AND re-encodes audio to AAC. AAC has ~21ms / 1024 samples of encoder lookahead which is normally compensated by the edit list atom (`edts`). With edit list disabled, that delay is audible as silent audio at the start → each short is 21-43ms out of sync. The error is small per clip but perceptually obvious.
**Why it's only shorts**: `longform_render.py` also uses `-use_editlist 0`, but it muxes audio ONCE over the whole concat so the AAC lookahead applies only once at t=0 — masked by natural silence. Shorts get re-cut per clip so every clip has the lookahead at its internal start time.
**Proposed fix options** (pick one):
1. **Remove `-use_editlist 0` from shorts only** — shorts go to YouTube/TikTok/IG/X where edit lists are fine. Spotify compliance only applies to longform.
2. **Pre-encode audio slice to AAC first, then stream-copy on mux** — bakes in the padding once during extract, not during final mux.
3. **Use `-af "aresample=async=1"` + explicit `-af "adelay"`** to compensate.
Recommendation: option 1 (smallest diff, matches platform expectations).

### 🟡 BUG-7: ffmpeg renders much slower than DaVinci Resolve — RESEARCH RETURNED

**Symptom**: User reports 30-90 min to render a 52-min 4K episode vs ~10-15 min in DaVinci. Says "ffmpeg is super suboptimal."

**Why DaVinci wins**: Metal-accelerated compute shaders for every filter stage + unified decode→filter→encode GPU pipeline. HEVC Main 10 decode uses M4's dedicated Media Engine. ffmpeg in our current config is **CPU-only for every filter** (lut3d, scale, hqdn3d, cas, eq, libass) — we're using ~10% of available silicon. The M4 has 4 P-cores + 6 E-cores + 40-shader GPU + separate Media Engine for codec; we fully saturate the P-cores and leave everything else idle.

**Verdict on alternatives** (don't waste time here):
- ❌ **DaVinci CLI** — doesn't exist. The Python Scripting API requires the Resolve GUI running.
- ❌ **HandBrake** — no LUT, no custom filter chain.
- ❌ **VapourSynth** — no maintained macOS Metal LUT filter; integration too complex.
- ❌ **MLT/Shotcut** — CPU-only for most filters on macOS, slower than current.
- ❌ **Build ffmpeg from master** — Homebrew 8.0.1 already has `scale_vt`, `tonemap_opencl`, videotoolbox decode. The 2025 Metal filters in master aren't production-stable.

**Action plan** (do these in order, benchmark each step):

1. **[BIGGEST WIN] Enable VideoToolbox hardware HEVC decode.** Prepend `-hwaccel videotoolbox -hwaccel_output_format videotoolbox` before `-ss` in the ffmpeg cmd, and prepend `hwdownload,format=p010le,` to the start of `vf_parts` so the frames get pulled back to RAM for the CPU filter chain. Media Engine runs concurrently with filter work → decode latency hidden inside filter time.
   - Test command to validate support:
     ```
     ffmpeg -hwaccel videotoolbox -hwaccel_output_format videotoolbox -i source_merged.mp4 -vf "hwdownload,format=p010le,format=yuv420p" -c:v h264_videotoolbox -q:v 65 -t 10 /tmp/test_vt.mp4
     ```
   - File: `agents/longform_render.py:_render_segment()` around line 276-280, and `agents/shorts_render.py:_render_short()` around line 236-249.

2. **Switch LUT interpolation tetrahedral → trilinear.** One-line config change: `lut_interpolation = "trilinear"` in `config.toml`. The agent says the quality diff is invisible at H.264 output quality. tetrahedral 3D interpolation is the most CPU-expensive filter after scale.

3. **Reduce worker count** from `min(os.cpu_count() // 2, 6)` → `min(os.cpu_count() // 4, 4)`. With hardware decode, 3-4 CPU filter pipelines can actually run without contention; 5 software-decode pipelines fight over P-cores + memory bus. File: `agents/longform_render.py` line 139.

4. **Disable or defang `hqdn3d` temporal.** Current: `hqdn3d=1.5:1.5:6:6`. Agent says: for talking heads, HEVC's own deblocking already removes mosquito noise, and temporal denoise on a moving person introduces subtle ghosting. Two options:
   - **Full disable**: `video_denoise = false` in config.
   - **Spatial only**: change `lib/encoding.py:get_video_polish_filters()` to emit `hqdn3d=1.5:1.5:0:0` (temporal = 0).

5. **[LARGER REFACTOR, DO LATER]** Consolidate longform + shorts into a single ffmpeg pass using `-filter_complex` with `split`/`tee`. Runs the LUT once, splits frames after, produces both outputs per decode. Cuts total filter work ~50%. This is the DaVinci architectural advantage replicated in ffmpeg. Non-trivial — skip until items 1-4 are done and benched.

### 🟡 BUG-7 — BENCHMARK RESULTS (2026-04-11, after compact)

Ran the 3 benchmarks on the real 4K HEVC 10-bit source (ep_2026-04-08_171445/source/DJI_20260408175419_0004_D.MP4, 2-minute slice, identity-scale 3840x2160):

| Test | Config | wall | fps | speedup |
|---|---|---|---|---|
| A | baseline: tetrahedral LUT + hqdn3d temporal=6 | 5:51 | 10 | 1.0x |
| B | trilinear LUT + hqdn3d temporal=0 | 5:16 | 11 | 1.11x |
| C | + `-hwaccel videotoolbox` HEVC decode | 5:05 | 12 | 1.15x |

**Verdict: DO NOT deploy.** The research agent's 3-5x prediction was wrong. Actual speedup is 15% total — not worth the quality tradeoff from trilinear LUT and dropping temporal denoise. The filter chain is the bottleneck, not decode or LUT interp. Hardware decode only adds 4% because the CPU filter chain is what's pegged, and VT decode just waits on it.

**What's probably actually slow** (hypotheses, not yet benchmarked):
1. **Identity-scale waste**: `scale=3840:2160:flags=lanczos+accurate_rnd+full_chroma_int` runs a full lanczos resample even when src_w==out_w. Should skip scale entirely at identity resolution. Estimated win: 10-20%.
2. **`format=yuv420p` 10-bit→8-bit conversion** happens on CPU with dither. Unavoidable for H.264 output but could be done via scale filter's dither once instead of a separate format filter.
3. **Parallel segment contention**: 6 workers × 10 fps = 60 fps aggregate, but memory/cache contention at 4K may mean 3-4 workers would hit the same throughput with less thrash.
4. **Item 5 from original plan still holds**: single ffmpeg pass with `split`/`tee` to render longform + shorts concurrently. Halves the filter work. Biggest potential win, biggest refactor.

**Action for next turn**: Don't touch production code for perf yet. If user wants to pursue faster rendering, investigate identity-scale skip first (small diff, benchmarkable in isolation). Otherwise accept current ~30-60 min render as the cost of quality and move on.

### 🟡 BUG-7 — ROUND 2: filter isolation benchmark (2026-04-11 post-compact)

Found the real bottleneck by isolating every filter on the same 4K 120s slice:

| Filter chain | wall | fps | marginal cost |
|---|---|---|---|
| null (decode+encode only) | 1:30 | 49 | — |
| + format=yuv420p | 1:29 | 49 | ~0 (free) |
| + lut3d tetrahedral | 2:17 | 22 | +48s (+54%) |
| + identity scale lanczos | 3:26 | 5 | +69s (+50%) |
| + hqdn3d temporal | 4:34 | 3 | +68s (+33%) |
| + cas sharpening | 5:35 | 2 | +61s (+22%) |
| + eq polish (full prod) | 6:06 | 2 | +31s (+9%) |

**Insights**:
1. **Identity lanczos scale costs 19% of total wall time** when src==out. BUT: only applies to `mode=None` segments (BOTH/NONE with zoom<=1.0). EP2 has zoom=1.2 on each speaker → 99% of segments use crop mode and actually need scale to upscale back. Still a free win for the handful of BOTH segments.
2. **hqdn3d adds 19%** — this is the biggest easy target. It's a quality filter the user turned on; dropping it is an A/B call.
3. **cas adds 17%** — same: user turned it on, dropping is an A/B call.
4. **eq adds 9%** — small marginal cost, could stay.
5. Dropping hqdn3d + cas + eq together = 44% speedup. Episode render time drops from ~45 min to ~25 min for EP2's 52 min source with 4 parallel workers.

**Applied fix (zero risk)**: identity-scale skip in `agents/longform_render.py:_get_crop_filter()`. When `mode is None and src_w == out_w and src_h == out_h`, return empty string and caller drops scale from the chain. Only helps BOTH segments on identity-resolution episodes. No quality impact.

**Pending user decision (task #40)**: hqdn3d, cas, eq — drop one/some/all? Recommendation for "worth it":
- `video_denoise = false` (drops hqdn3d): -19% render time. Quality: may show HEVC mosquito noise in dark areas. Most visible on tripod-locked talking heads.
- `video_sharpen = false` (drops cas): -17% render time. Quality: video slightly softer, pore/skin detail reduced.
- `video_polish = false` (drops eq): -9% render time. Quality: loses the subtle saturation/gamma lift — but this is so subtle you probably won't notice.

Safest speed win after identity-scale: set `video_polish = false` in config for a free 9% with nearly invisible quality loss. If user wants more speed, disable `video_denoise` (biggest win, most visible tradeoff).
